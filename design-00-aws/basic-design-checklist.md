## 3. ネットワーク設計

### 3.1 VPC設計

- [ ] VPC CIDRは将来の拡張を考慮した `/16` で確保
- [ ] 既存VPCとのピアリング・TGW接続時にCIDR重複がないか確認
- [ ] サブネットを3層分離（Public / Private / DB）
- [ ] マルチAZ構成（本番3AZ、開発2AZ）

### 3.2 サブネット設計

- [ ] サブネットCIDRは `/24` 単位で整理し管理しやすくする
- [ ] 各サブネットのAZ・用途を一覧化した
- [ ] Internet Gateway をPublicサブネットに設定
- [ ] NAT Gateway はPrivateサブネットの外部通信用に配置
- [ ] 本番環境はNAT GWをAZごとに冗長化
- [ ] ルートテーブルの設定漏れがないか確認

### 3.3 VPC Endpoint

- [ ] ECR（ecr.api / ecr.dkr）
- [ ] S3（Gateway型）
- [ ] CloudWatch Logs（logs）
- [ ] SSM / SSM Messages（踏み台不要化）
- [ ] Secrets Manager（必要な場合）

### 3.4 セキュリティグループ設計

- [ ] 用途ごとにSGを分離（ALB用 / ECS用 / RDS用 / Cache用）
- [ ] インバウンドは最小ポートのみ許可
- [ ] SGのソース指定はIPではなく **SGのID参照** を使う
- [ ] アウトバウンドの制限が必要か検討（デフォルト全開放でよいか）

## 4. コンピューティング設計

### 4.1 ECS / Fargate

- [ ] クラスター名・サービス名の命名規則を決定
- [ ] タスク定義のCPU / Memoryサイズを適切に設定
- [ ] コンテナイメージはECRを使用（DockerHub直接参照は避ける）
- [ ] ログドライバーは `awslogs` でCloudWatch Logsへ送信
- [ ] ヘルスチェックパスをALBとタスク定義の両方に設定
- [ ] Desired Count は本番2台以上（単一障害点を排除）
- [ ] デプロイ方式（Blue/Green等）を決定

### 4.2 Auto Scaling

- [ ] スケーリングポリシーの種類を選定（ターゲット追跡 推奨）
- [ ] CPU使用率70%をスケールアウト閾値の目安に設定
- [ ] Min / Max Capacity を環境ごとに設定
- [ ] スケールイン時のタスク保護（draining）を確認

### 4.3 タスクロール設計

- [ ] タスク実行ロールとタスクロールを分離
- [ ] 各ロールの権限（ECR Pull, Logs, Secrets Manager等）を最小化

### （Lambda使用時：補足）

- [ ] メモリサイズとタイムアウト値を適切に設定
- [ ] VPC内配置の要否を判断（RDS接続等）
- [ ] 同時実行数の上限設定（Concurrency Limit）
- [ ] デッドレターキュー（DLQ）の設定

---

## 5. データ層設計

### 5.1 RDS / Aurora

- [ ] エンジン選定（MySQL / PostgreSQL）とバージョン固定
- [ ] Multi-AZ or Auroraクラスター（Writer + Reader）
- [ ] DBサブネットグループをDB専用サブネットに設定
- [ ] パラメータグループをカスタム作成（デフォルト変更不可のため）
- [ ] 自動バックアップの保持期間（本番7日以上）
- [ ] メンテナンスウィンドウを業務時間外に設定
- [ ] 削除保護（Deletion Protection）を有効化
- [ ] 暗号化（KMS）を有効化

### 5.2 S3

- [ ] バケット名のグローバル一意性確認
- [ ] バケットごとの用途を一覧化（assets / logs / terraform-state等）
- [ ] バージョニング有効化（本番必須）
- [ ] パブリックアクセスブロックを全項目有効化
- [ ] ライフサイクルルール設定（不要オブジェクトの自動削除）
- [ ] サーバーサイド暗号化（SSE-S3 or SSE-KMS）
- [ ] バケットポリシーで最小権限を設定

### （ElastiCache使用時：補足）

- [ ] Redis / Memcachedの選定
- [ ] クラスターモード or シングルノードの決定
- [ ] サブネットグループをPrivateサブネットに設定
- [ ] 保存時・転送時の暗号化を有効化

## 6. セキュリティ設計

### 6.1 IAM設計方針

- [ ] ルートアカウントのMFA有効化・アクセスキー削除
- [ ] IAMユーザーは個人ごとに発行（共有アカウント禁止）
- [ ] サービス間通信はIAMロール使用（アクセスキー禁止）
- [ ] ポリシーは管理ポリシー優先、インラインポリシーは最小化
- [ ] 権限境界（Permission Boundary）の要否検討

### 6.2 シークレット管理

- [ ] DBパスワード → Secrets Manager（自動ローテーション有効化）
- [ ] 環境変数の機密情報 → Parameter Store（SecureString）
- [ ] TLS証明書 → ACM（自動更新）
- [ ] アプリ側でシークレットをハードコードしていないか確認

### 6.3 WAF設計

- [ ] ALB or CloudFrontにWAFをアタッチ
- [ ] AWSマネージドルール（Core Rule Set）を最低限有効化
- [ ] SQLインジェクション / 既知の悪意あるリクエストルール有効化
- [ ] レートベースルール（同一IP過剰リクエスト制限）
- [ ] WAFログをS3またはCloudWatch Logsへ出力

### 6.4 セキュリティサービス

- [ ] GuardDuty有効化（脅威検知）
- [ ] CloudTrail有効化（操作ログ、S3保存）
- [ ] Config有効化（リソース変更履歴）
- [ ] SecurityHub有効化（セキュリティスコア可視化）
- [ ] Inspector有効化（脆弱性スキャン、本番環境）

### （暗号化：横断補足）

- [ ] 保存時暗号化：RDS / S3 / EBS / CloudWatch Logs
- [ ] 転送時暗号化：ALBでHTTPS終端、ACM証明書設定
- [ ] KMSカスタムキーの要否（コンプライアンス要件確認）

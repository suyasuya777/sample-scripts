## 4. 非機能要件

### 4.1 可用性要件

| 項目 | 要件 | 根拠・備考 |
|------|------|-----------|
| 目標稼働率 | 【例：99.9%】 | SLA契約に基づく／社内基準 |
| 計画停止の許容 | 【例：月1回、深夜帯2時間まで許容】 | メンテナンスウィンドウ確保のため |
| 障害時の縮退運転 | 【例：DB障害時も静的コンテンツは配信継続】 | 必要に応じて記載 |

### 4.2 性能・拡張性要件

| 項目 | 要件 | 根拠・備考 |
|------|------|-----------|
| 想定同時接続数（通常時） | 【】 | |
| 想定同時接続数（ピーク時） | 【】 | セール時は通常の◯倍を想定 |
| レスポンスタイム目標 | 【例：API平均500ms以内】 | ユーザー体験要件 |
| スループット要件 | 【例：◯件/秒のリクエスト処理】 | |
| スケーラビリティ方針 | 【例：トラフィック増加時に自動スケールすること】 | Auto Scaling前提 |
| 将来の利用者数予測 | 【例：1年後に現在の3倍を想定】 | キャパシティプランニング用 |

### 4.3 信頼性要件（RPO/RTO）

| 項目 | 要件 | 根拠・備考 |
|------|------|-----------|
| RPO（目標復旧時点） | 【例：1時間以内】 | 許容できるデータ損失量 |
| RTO（目標復旧時間） | 【例：4時間以内】 | 障害発生から復旧までの許容時間 |
| バックアップ世代数 | 【例：直近7日分】 | |
| DR要否 | 【あり/なし】 | ありの場合、別リージョン構成が必要 |

### 4.4 セキュリティ要件

| 項目 | 要件 | 根拠・備考 |
|------|------|-----------|
| 通信の暗号化 | 全通信をTLS 1.2以上で暗号化 | |
| データの暗号化（保存時） | 個人情報・機密情報を含むデータストアは暗号化必須 | |
| アクセス制御 | 管理画面は社内IP/VPN経由のみアクセス許可 | |
| 認証方式 | 【例：社内SSO連携】 | |
| 監査ログ | 全AWS API操作のログを1年間保持 | 内部統制要件 |
| 脆弱性対策 | コンテナイメージの脆弱性スキャンを必須とする | |
| Webアプリ攻撃対策 | WAFによるOWASP Top 10対策を実施 | |
| 準拠すべき法令・基準 | 【例：個人情報保護法、PCI DSS（決済情報を扱う場合）】 | |

### 4.5 運用・保守要件

## 7. 受け入れ基準

### 7.1 完了の定義（Definition of Done）

| No. | 基準 |
|-----|------|
| 1 | 全環境（dev/stg/prod）でインフラがTerraformコードにより再現可能であること |
| 2 | 第4章で定義した非機能要件を満たしていることを試験で確認できること |
| 3 | セキュリティ要件（4.4）に関するチェックリストが全項目クリアしていること |
| 4 | 監視・アラート設定が稼働し、Slack通知が正常に届くことを確認したこと |
| 5 | 運用手順書・障害対応手順書が整備されていること |

### 7.2 検証方法

| 検証項目 | 検証方法 | 実施環境 |
|---------|---------|---------|
| 性能要件 | 負荷試験ツール（例：JMeter）によるピーク負荷再現 | stg |
| 可用性要件 | AZ障害を模擬したフェイルオーバー試験 | stg |
| セキュリティ要件 | 脆弱性診断、WAFルール動作確認 | stg/prod |
| バックアップ・リストア | スナップショットからのリストア試験 | dev/stg |
| 監視・アラート | 意図的に閾値超過させアラート発火を確認 | stg |

## 補足：機能要件・非機能要件について

### 機能要件

システムが「何をするか」を示す要件

**インフラ基盤が提供すべき機能**

| 項目 | 内容 |
|------|------|
| サーバー | 台数・タイプ（CPU/メモリ/ディスク性能、インスタンスタイプ） |
| ネットワーク構成 | VPC/サブネット、パブリック/プライベート、VPN/Direct Connect要否 |
| ストレージ要件 | ブロックストレージ、オブジェクトストレージ、バックアップ/スナップショットの頻度・保存期間 |
| ミドルウェア | OSイメージ、コンテナ基盤、ロードバランサ、キャッシュ、メッセージキュー |
| 運用自動化 | Terraform/CloudFormationテンプレート、Ansibleプレイブック、CI/CDパイプライン |

| 項目 | 内容 |
|------|------|
| 実施タイミング | 要件定義フェーズ |
| ドキュメント | インフラ・ユースケース図 ⇒ ネットワーク利用シーン別トポロジ図<br>コンポーネント構成図（サーバ、データベース、ロードバランサ配置図）<br>機能仕様書 ⇒ インフラ要件仕様書 |
| 測定方法 | Terraform applyの成功<br>疎通確認<br>ポート開放確認 |

---

### 非機能要件

システムが「どうあるべきか」を示す要件

**性能・品質・運用性など品質特性**

| 項目 | 内容 |
|------|------|
| 性能・スケーラビリティ | レイテンシー、スループット、同時接続数、Auto Scaling条件 |
| 可用性・冗長性 | SLA設定、マルチAZ/マルチリージョン、フェールオーバー設計 |
| セキュリティ | SG/NACL、IAMポリシー、暗号化、監査ログの集中管理 |
| 運用・監視 | CloudWatch／Prometheus メトリクス、ログ集約、アラート設計、運用手順 |
| コスト | 課金モデル、リソース最適化、Reserved Instances／Savings Plans 利用 |
| コンプライアンス | PCI-DSS（クレジットカード）／SOC2（顧客データ）等の認証要件 |

| 項目 | 内容 |
|------|------|

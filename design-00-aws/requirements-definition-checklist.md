# インフラ要件定義チェックリスト（AWS版）

> 基本設計・詳細設計のチェックリストが「実現方式・パラメータの妥当性」を確認するのに対し、  
> 本チェックリストは **「ビジネス要求が漏れなく言語化され、後続の設計に渡せる状態か」** を確認します。

---

## 1. 背景・目的

### 1.1 プロジェクトの背景

- [ ] システム・インフラが必要になったビジネス上の背景を記載した
- [ ] 背景がステークホルダー間で共通認識になっているか確認した

### 1.2 目的

- [ ] プロジェクトの目的を箇条書きで明確化した
- [ ] 各目的が後続の非機能要件（4章）と紐づいているか確認した

### 1.3 本書の位置付け

- [ ] 要件定義書・基本設計書・詳細設計書の役割分担を明記した
- [ ] 「どう実現するか」の記述が本書に混入していないか確認した（設計書側の記述になっていないか）

---

## 2. プロジェクト概要

### 2.1 対象システム概要

- [ ] システム名・概要・利用者規模を記載した
- [ ] 利用時間帯（24時間365日 or 業務時間内等）を明記した
- [ ] 想定ピーク時期・タイミングを記載した

### 2.2 ステークホルダー

- [ ] 全役割（発注者/PM/インフラ/アプリ/セキュリティ/運用）の担当者を確定した
- [ ] 要件承認者（決裁者）を明確にした

### 2.3 開発・リリーススケジュール概要

- [ ] 要件定義〜本番リリースまでの主要マイルストーンを設定した
- [ ] スケジュールが後続の設計・構築・試験期間として現実的か確認した

### 2.4 予算・契約条件

- [ ] 月額運用予算の上限を確定した
- [ ] AWSアカウントの契約形態（Organizations配下/個別）を確認した
- [ ] 既存契約・既存リソースとの共用の有無を確認した

---

## 3. 業務要件（システム要求・機能要件）

### 3.1 提供機能概要（機能要件）

- [ ] インフラ構成に影響する主要機能を洗い出した
- [ ] バッチ処理・非同期処理等、常駐Webアプリ以外のワークロードの有無を確認した
- [ ] アプリケーション要件定義書との役割分担・参照関係を明記した

### 3.2 外部連携要件

- [ ] 連携先システムごとに連携方式・通信方向・データ内容を一覧化した
- [ ] 固定IPアドレスが必要な連携先の有無を確認した
- [ ] 外部連携が原因でNAT Gateway/専用線等の追加要件が発生しないか確認した

### 3.3 利用者属性・アクセス経路

- [ ] エンドユーザー・社内管理者・開発者ごとのアクセス経路を整理した
- [ ] 管理画面等、公開範囲を限定すべき機能の有無を確認した
- [ ] 想定人数/台数が後続のキャパシティ設計（4.2）と整合しているか確認した

---

## 4. 非機能要件

### 4.1 可用性要件

- [ ] 目標稼働率（SLA）を数値で確定した
- [ ] 計画停止（メンテナンスウィンドウ）の許容範囲を確認した
- [ ] 障害時の縮退運転要否・範囲を検討した

### 4.2 性能・拡張性要件

- [ ] 通常時・ピーク時の同時接続数を見積もった
- [ ] レスポンスタイム目標を機能ごとに設定した（必要に応じて）
- [ ] スループット要件（リクエスト数/秒等）を確認した
- [ ] 将来（1年後等）の利用者数予測を記載した
- [ ] スケーラビリティ方針（自動スケール前提か）を明記した

### 4.3 信頼性要件（RPO/RTO）

- [ ] RPO・RTOの目標値を業務影響を踏まえて確定した
- [ ] バックアップ世代数の要件を確認した
- [ ] DR要否を確定した（要の場合、別リージョン構成のコスト・工数を関係者と合意）

### 4.4 セキュリティ要件

- [ ] 通信・保存データの暗号化要件（TLSバージョン、暗号化対象）を確認した
- [ ] アクセス制御方針（管理画面のIP制限等）を確認した
- [ ] 認証方式（SSO連携等）の要否を確認した
- [ ] 監査ログの保持期間・対象範囲を確認した
- [ ] 脆弱性対策（イメージスキャン等）の要件を確認した
- [ ] WAF等によるWebアプリ攻撃対策の要件を確認した
- [ ] 準拠すべき法令・基準（個人情報保護法、PCI DSS等）を洗い出した

### 4.5 運用・保守要件

- [ ] 監視体制（時間帯別の対応レベル）を確認した
- [ ] アラート通知方法（Slack/電話等）を確認した
- [ ] ログ保持期間（アプリログ/監査ログ）を確認した
- [ ] 運用担当者のアクセス権限方針（ロールベース分離）を確認した
- [ ] メンテナンス窓口の曜日・時間帯を確認した
- [ ] 障害時のエスカレーションフローの要否・概要を確認した

### 4.6 移行要件（既存システムからの移行がある場合）

- [ ] 移行対象データの種類・容量を見積もった
- [ ] 移行方式（オンライン/オフライン）を検討した
- [ ] 移行時の許容停止時間を業務側と合意した
- [ ] 並行稼働期間の要否を確認した
- [ ] ロールバック方針（切り戻し手順の要否）を確認した
- [ ] 移行要件が「該当なし」の場合、その旨を明記した

### 4.7 コスト要件

- [ ] 月額運用コストの上限を確定した
- [ ] コスト最適化方針（開発環境の夜間停止等）を確認した
- [ ] コスト可視化要件（プロジェクト/環境別按分）を確認した

### 4.8 法令・組織standards準拠要件

- [ ] 利用リージョンの制約（データ国内保存等）を確認した
- [ ] 参照すべき社内セキュリティ基準・ガイドラインを特定した
- [ ] 外部監査対応の要否・頻度を確認した

---

## 5. システム環境要件

### 5.1 環境構成要件

- [ ] dev/stg/prodの3環境が必須であることを確認した
- [ ] DR環境の要否を確定した（4.3のDR要否と整合しているか確認）

### 5.2 クラウドプラットフォーム要件

- [ ] クラウドプロバイダ・主要リージョンを確定した
- [ ] 既存AWS環境（Organizations等）との関係を確認した
- [ ] マルチクラウド要否を確認した

### 5.3 ネットワーク要件

- [ ] 既存社内ネットワークとの接続要否（VPN/Direct Connect）を確認した
- [ ] インターネット公開要否・公開範囲を確認した
- [ ] 既存VPCとのCIDR重複回避の制約有無を確認した
- [ ] 利用予定ドメインを確認した

### 5.4 開発・運用ツール要件

- [ ] ソースコード管理ツールを確定した
- [ ] CI/CDツール・IaCツールを確定した
- [ ] 監視・通知ツールを確定した
- [ ] チケット管理ツールを確認した（既存ツールとの統合要否含む）

---

## 6. 制約条件・前提条件

### 6.1 制約条件

- [ ] 法令・組織上の制約を全て洗い出した
- [ ] 各制約の理由・根拠を明記した
- [ ] 制約が後続の設計（リージョン選定、IaC方針等）に正しく反映される見込みであることを確認した

### 6.2 前提条件

- [ ] AWSアカウント準備状況を確認した
- [ ] アプリケーションのコンテナ化状況を確認した
- [ ] その他の前提条件（既存資産の利用可否等）を洗い出した

### 6.3 リスク・課題

- [ ] 未確定事項（トラフィック量、DR要件等）をリスクとして一覧化した
- [ ] 各リスクの影響度を評価した
- [ ] 各リスクの対応方針・解消予定時期を設定した

---

## 7. 受け入れ基準

### 7.1 完了の定義（Definition of Done）

- [ ] 全環境がIaC（Terraform等）で再現可能であることを完了条件に含めた
- [ ] 第4章の非機能要件が試験で確認可能な形になっているか確認した
- [ ] セキュリティ要件のチェックリストとの紐付けを確認した
- [ ] 監視・アラートの動作確認を完了条件に含めた
- [ ] 運用手順書・障害対応手順書の整備を完了条件に含めた

### 7.2 検証方法

- [ ] 性能要件の検証方法（負荷試験ツール・実施環境）を確定した
- [ ] 可用性要件の検証方法（フェイルオーバー試験）を確定した
- [ ] セキュリティ要件の検証方法（脆弱性診断等）を確定した
- [ ] バックアップ・リストア試験の実施環境・手順を確定した
- [ ] 監視・アラートの試験方法を確定した

---

## 8. 用語集

- [ ] 要件定義書内で使用した略語・専門用語を網羅的に記載した
- [ ] 基本設計書・詳細設計書の用語集との重複・矛盾がないか確認した

## 全体整合性チェック

### 後続文書との対応確認

- [ ] 4章の各非機能要件項目が、基本設計書のいずれかの章で「実現方式」として具体化される見込みであることを確認した
- [ ] 6.3のリスク・課題が、プロジェクト進行中に追跡管理される仕組み（課題管理表等）に登録されている
- [ ] 本書の数値（稼働率、RPO/RTO、コスト上限等）が基本設計書・詳細設計書の該当箇所と矛盾していない

### レビュー・承認

- [ ] 事業責任者によるレビュー・承認を受けた
- [ ] セキュリティ担当によるレビューを受けた（4.4, 4.8）
- [ ] 運用担当によるレビューを受けた（4.5）
